# GOAT — Decentralized Agentic Data-Enrichment Protocol

A Phase-1 MVP where two autonomous agents coordinate to enrich data and settle
payment on-chain. A **Worker** agent uses a lightweight LLM to convert messy,
unstructured CSV rows into clean JSON; a **Master** agent verifies that JSON
deterministically (no LLM) and, on success, pays the Worker an x402-style
micro-payment on Base Sepolia. Both agents carry ERC-8004 on-chain identities.

Built for the GOAT AI Builder Grant. Testnet only.

## Run & Operate

The protocol lives in the `agent-protocol/` package.

- `pnpm --filter @workspace/agent-protocol run compile` — compile the Solidity contracts (solc → `build/`)
- `pnpm --filter @workspace/agent-protocol run deploy` — deploy contracts to Base Sepolia, register both agents, mint + approve mUSDC, write `deployed-addresses.json`
- `pnpm --filter @workspace/agent-protocol run start` — run the enrichment + settlement loop over `data/raw.csv`
- `pnpm --filter @workspace/agent-protocol run typecheck` — typecheck the package

Required to go LIVE on-chain:
- `DEPLOYER_PRIVATE_KEY` (secret) — a **testnet-only** wallet that acts as the Master/deployer. Fund it with a little Base Sepolia ETH from a faucet.
- `RPC_URL` (optional) — defaults to `https://sepolia.base.org`.

LLM access uses the Replit OpenAI AI Integration (`AI_INTEGRATIONS_OPENAI_BASE_URL` / `AI_INTEGRATIONS_OPENAI_API_KEY`, auto-provisioned). No user OpenAI key needed.

Without `DEPLOYER_PRIVATE_KEY` + a deployment, the harness runs in **SIMULATED** mode: it still enriches and verifies, and logs the settlement it *would* make.

## Stack

- Node.js 24 + TypeScript, pnpm workspace package `@workspace/agent-protocol`
- Contract tooling: `solc` (compile-on-demand) + `viem` (deploy/interact) — deliberately **not** Hardhat
- Chain: Base Sepolia (chainId 84532)
- LLM: Replit OpenAI AI Integration, model `gpt-5-nano` (override via `WORKER_MODEL`)

## Where things live

- `agent-protocol/contracts/` — `IdentityRegistry.sol` (ERC-8004 identities), `MockUSDC.sol` (6-decimal settlement token), `Settlement.sol` (x402-style settle + replay guard)
- `agent-protocol/scripts/` — `compile.ts`, `deploy.ts`
- `agent-protocol/src/` — `worker.ts` (LLM enrichment), `master.ts` (verify + settle), `schema.ts` (target shape + deterministic validator), `chain.ts` (viem clients, addresses), `harness.ts` (orchestration loop, entrypoint)
- `agent-protocol/data/raw.csv` — the 5-row trigger dataset
- `agent-protocol/deployed-addresses.json` — generated by deploy (gitignored; contains the Worker's testnet key)

## Architecture decisions

- **solc + viem over Hardhat:** avoids a heavy toolchain and a persistent local node, keeping memory flat on a mobile cloud instance. Compilation runs on demand and exits.
- **Single funder:** the deployer is the Master; the Worker is a generated testnet keypair that only *receives* tokens, so only one account needs faucet ETH. The registry allows registering an address on another's behalf to support this.
- **Honest x402 framing:** real on-chain ERC-20 settlement to the Worker's identity, run in-process. Simplification vs. the full spec: no external facilitator and a direct `transferFrom` rather than a relayed EIP-3009 `transferWithAuthorization`.
- **Deterministic verification:** the Master uses pure rule-based checks (required non-empty fields + email regex) — never an LLM — so payment is gated on objective verification. Row 5 (no email) is correctly withheld.

## Other artifacts

- `artifacts/api-server`, `artifacts/mockup-sandbox` — scaffold from the base template; not part of the protocol MVP.

## User preferences

- Ruthless compute efficiency: process rows sequentially, lightweight model, no persistent local chain. Avoid choices that risk crashing a mobile Replit instance.
- Scope discipline: Phase 1 excludes ClawUp, agent scaling, multi-worker fan-out, mainnet, and a web frontend.

## Gotchas

- The public Base Sepolia RPC can lag on reads immediately after a write — read-back of just-written contract state may return stale/zero values. `deploy.ts` retries `agentId` reads to handle this.
- Re-running `start` reuses the deployment; each run uses a fresh `runId` in the jobId so the Settlement replay guard doesn't reject repeat demos.
